# AGENTS.md

本仓库用于建设 `Luna` Avalonia 控件库。后续所有代码、文档、主题和示例都应围绕统一视觉语言、统一控件 API，以及桌面端与移动端一致体验展开。

## 仓库定位

- `Luna` 是面向 Avalonia 的控件库工作区，当前按桌面端与移动端拆分实现与示例。
- `src/Luna.Desktop/` 负责桌面端控件、主题与资源。
- `src/Luna.Mobile/` 负责移动端控件、主题与资源。
- `samples/Luna.Desktop.Sample/` 与 `samples/Luna.Mobile.Sample/` 用于控件调试、交互验证和示例展示。
- `docs/` 统一保存控件规划、组件分析、阶段拆分和使用文档。
- TDesign Mobile 是移动端组件语义和交互设计的主要参考；分析需求时先对齐组件语义，再判断 Avalonia 原生能力与 Luna 自定义控件边界。

## 总体原则

- 优先服务 `Luna` 的长期目标：统一样式、统一 API、跨端一致、可持续扩展。
- 先读现有实现再新增抽象。新增控件、主题资源或示例前，先查看相邻控件、主题字典和 sample 页面。
- 不把桌面端专属能力写死进公共控件 API。窗口、右键菜单、悬停、文件系统等差异应隔离在平台实现、样式变体或宿主示例中。
- 控件库代码必须保持可换肤、可绑定、可测试。业务流程不要进入控件层。
- 不做无关重排、批量格式化或目录搬迁。涉及主题 token、基础控件、共享样式和构建配置时，先评估影响面。

## 目录职责

### `src/Luna.Desktop/`

- 放置桌面端控件实现、桌面主题和桌面资源。
- 优先复用 Avalonia 现有控件与主题机制，只在桌面端交互语义明显不足时增加自定义控件。
- 桌面端实现应关注键鼠效率、焦点可见性、窗口缩放、高密度布局和快捷操作。

### `src/Luna.Mobile/`

- 放置移动端控件实现、移动主题和移动资源。
- 移动端优先保证触控目标尺寸、手势行为、安全区域、虚拟键盘适配和窄屏布局稳定性。
- 与 TDesign Mobile 对齐时，优先沉淀语义一致的控件 API、状态定义和主题 token，不直接照搬前端实现细节。

### `samples/`

- 只承担示例、演示和验证，不承载控件库公共逻辑。
- 示例页面应覆盖默认用法、关键状态、数据绑定、禁用态和异常态。
- 示例数据应内置，不依赖外部服务。

### `docs/`

- 统一维护规划、分析、设计决策和使用文档。
- 当前优先维护的入口文档：
  - `docs/README.md`
  - `docs/analysis/tdesign-mobile-component-analysis.md`
  - `docs/planning/luna-control-roadmap.md`
  - `docs/controls/README.md`
- 不要把规划类文档散落到仓库根目录或 sample 目录。

## Avalonia 控件开发规则

- 开发前先判断控件类型：
  - Avalonia 已有成熟控件且语义匹配时，优先做主题皮肤、资源 token 和样式封装。
  - Avalonia 可组合实现但缺少目标语义时，提供 Luna 包装控件，内部复用基础控件。
  - 需要手势、弹层状态机、复杂选择器、自绘或平台服务时，再开发 Luna 自定义控件。
- 可复用控件优先使用 `TemplatedControl`；固定结构、明确组合关系的局部组件再考虑 `UserControl`。
- 需要参与样式、资源、绑定和状态传播的值优先使用 `StyledProperty<T>`；仅在高频内部状态且不需要样式系统参与时考虑 `DirectProperty`。
- CLR 属性包装器必须与 Avalonia 属性同名，默认值、绑定模式和变更回调应明确。
- 模板部件在 `OnApplyTemplate` 中获取；替换模板引用前先解绑旧事件，再绑定新事件。
- 视觉状态优先通过伪类、资源和样式选择器表达，不在 code-behind 中直接硬改可视树外观。
- 自定义布局必须正确实现 `MeasureOverride` 与 `ArrangeOverride`，避免带副作用的测量逻辑。
- 输入密集控件需要覆盖焦点、键盘导航、指针捕获、取消路径、禁用态和触控场景。

## 主题与设计约束

- 主题应围绕 token 组织：颜色、字号、间距、圆角、边框、阴影、尺寸和动效。
- 同一控件至少考虑普通、悬停、按下、焦点、禁用、选中、错误、加载和空状态。
- 深色与浅色主题都必须保持可读，不允许只针对单一主题修样式。
- 默认外观应写入主题或资源字典，不要把视觉参数散落到控件构造函数。
- 新样式要保持统一而克制，避免为单个控件制造孤立视觉语言。

## 文档与示例要求

- 新控件应同时补齐最小示例和对应文档。
- 文档至少说明控件定位、核心 API、关键状态、主题资源键、跨端注意事项和验证方式。
- 若新增或调整 TDesign 映射，同步更新 `docs/analysis/tdesign-mobile-component-analysis.md`。
- 若新增阶段任务、优先级或验收标准，同步更新 `docs/planning/luna-control-roadmap.md`。

## 验证要求

完成改动后优先执行与影响范围匹配的最小验证：

- 全仓基础验证：
  - `dotnet restore Luna.sln`
  - `dotnet build Luna.sln --no-restore`
- 桌面端示例验证：
  - `dotnet run --project samples/Luna.Desktop.Sample/Luna.Desktop.Sample.csproj`
- 移动端示例或移动库验证：
  - `dotnet build samples/Luna.Mobile.Sample/Luna.Mobile.Sample.csproj --no-restore`
- UI 改动需要说明手测范围：启动、导航、主题切换、窗口缩放、主要交互、键盘操作和窄屏表现。
- 如果无法完成验证，结果说明中必须写清原因、未验证项和剩余风险。

## 协作约束

- 开始任务前先确认影响范围属于 `Desktop`、`Mobile`、`Samples`、`Docs` 还是共享主题资源。
- 根目录只放工作区级说明、solution 和共享配置，不在根目录堆放临时设计稿或零散方案文档。
- 不删除用户未明确要求删除的示例、资源、配置或生成文件。
- 多人或多 agent 并行时，尽量按模块隔离修改；共享主题、基础控件和构建配置要避免互相覆盖。
- 代码评审优先关注绑定失效、模板部件泄漏、主题回归、布局溢出、平台假设、线程阻塞、命令状态错误和缺失验证。
